# DLP Proxy v2.0 — Архитектура

## 1. Умная система обнаружения ДСП

### Подход 1 — Словари + двухуровневый контекст

**Как работает сейчас (v1):**
```
Текст → regex → нашли ИНН? → БЛОК!
```
Проблема: 59 срабатываний на "ФИО" в обычном Word-файле.

**Как будет работать (v2):**
```
Текст → 1) Найти потенциальные данные (кандидаты)
       → 2) Проверить контекст (300 символов вокруг)
       → 3) Проверить комбинации (ФИО + телефон = ДСП, ФИО одно = нет)
       → 4) Рассчитать SCORE (баллы риска)
       → Если score >= порог → ФЛАГ (не блок!)
```

**Уровни данных:**
| Тип | Один = ДСП? | С контекстом = ДСП? | Примеры |
|-----|-------------|---------------------|---------|
| Гриф | ДА | — | "для служебного пользования", "совершенно секретно" |
| Идентификатор | НЕТ | ДА (+ ФИО рядом) | СНИЛС, ИНН, паспорт, номер карты |
| ПДн-маркер | НЕТ | ДА (+ идентификатор) | ФИО, телефон, email, адрес |
| Финансы | НЕТ | ДА (+ реквизиты) | Номер счёта, БИК |

**Комбинации которые вызывают флаг:**
- ФИО + паспорт → ВЫСОКИЙ
- ФИО + телефон + email → ВЫСОКИЙ
- ФИО + адрес → СРЕДНИЙ
- Телефон один → НЕТ
- ФИО одно → НЕТ (слишком много ложных)
- Гриф один → ВЫСОКИЙ (всегда)

### Подход 2 — NER (именованные сущности)

Библиотека **Natasha** (русский язык, работает локально):
```python
from natasha import Doc, Segmenter, NewsNERTagger, NewsEmbedding
# Распознаёт: PER (люди), LOC (места), ORG (организации)
# Отличает "Иванов Иван" от "Красная Площадь"
```

Добавляется как **дополнительный слой** поверх regex — для валидации ФИО.

## 2. Система прозрачности

### Модель работы

```
Сотрудник загружает файл в DeepSeek
     ↓
DLP анализирует → находит ДСП-данные
     ↓
Файл НЕ блокируется, но:
  → Создаётся ИНЦИДЕНТ (event)
  → Админу: уведомление в панели "🔴 Новый инцидент"  
  → Сотруднику: уведомление в клиенте "⚠ Файл помечен"
     ↓
Админ видит: файл, откуда, куда, какие правила сработали
Админ НЕ видит содержимое файла (приватность)
     ↓
Админ нажимает "Запросить доступ к содержимому"
  → Сотруднику приходит запрос в клиенте
  → Сотрудник нажимает "Разрешить" или "Отклонить"
     ↓
Если разрешено → Админ видит содержимое файла
  → Админ решает: ложное срабатывание или реальная утечка
  → Может написать сотруднику в чат
```

### Типы событий

```
incident    — DLP нашла подозрительные данные
upload      — обычная загрузка файла (без ДСП)
download    — скачивание файла
access_req  — админ запросил доступ к содержимому
access_ok   — сотрудник разрешил просмотр
access_deny — сотрудник отклонил
message     — сообщение в чат
```

### API (HTTP эндпоинты)

**Для клиента:**
```
GET  /api/notifications?client_ip=...   → список уведомлений
POST /api/access_response               → ответ на запрос доступа
GET  /api/messages?client_ip=...        → сообщения чата
POST /api/messages                      → отправить сообщение
```

**Для админа (в панели):**
```
POST /admin/access_request    → запросить доступ к файлу
GET  /admin/incidents         → список инцидентов
GET  /admin/chat?ip=...       → чат с сотрудником
POST /admin/chat/send         → отправить сообщение
```

### Хранение

Файл `logs/transparency.json`:
```json
{
  "incidents": [...],
  "access_requests": {
    "event_id": {
      "requested_by": "admin",
      "requested_at": "2026-03-20T16:00:00",
      "status": "pending|approved|denied",
      "responded_at": null
    }
  },
  "messages": {
    "192.168.15.41": [
      {"from": "admin", "text": "...", "time": "..."},
      {"from": "client", "text": "...", "time": "..."}
    ]
  }
}
```
